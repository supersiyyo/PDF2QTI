# Deployment & Migration Guide

This guide details the steps required to migrate the Doc-to-Quiz application from a local development environment to production, utilizing **Railway** for the backend and **SiteGround** for the frontend.

## 1. Backend Deployment (Railway)

Railway provides a seamless deployment experience for Python applications by automatically detecting the environment via Nixpacks and installing dependencies from `requirements.txt`.

### Preparation Steps

1. **Configure CORS:**
   In `backend/main.py`, update the `CORSMiddleware` configuration to explicitly allow requests from your production domain: **https://csun.sose.dev**.

   ```python
   app.add_middleware(
       CORSMiddleware,
       allow_origins=["https://csun.sose.dev"],
       allow_credentials=True,
       allow_methods=["*"],
       allow_headers=["*"],
   )
   ```

2. **Define Start Command:**
   Railway needs to know how to start the application and bind to the port it dynamically assigns. Create a file named `Procfile` in the `backend/` directory with the following content:
   ```text
   web: uvicorn main:app --host 0.0.0.0 --port $PORT
   ```
   _(Alternatively, this can be set as a Custom Start Command in the Railway dashboard)._

### Deployment Steps

1. Push your repository to GitHub.
2. Log in to Railway and create a new project.
3. Choose "Deploy from GitHub repo" and select your Doc-to-Quiz repository.
4. Go to the Service Settings and configure the **Root Directory** to `/backend`.
5. Navigate to the Variables tab and add your `GEMINI_API_KEY`.
6. Railway will automatically build and deploy the service. Once deployed, note the Public Networking URL generated for the backend (e.g., `https://pdf2qti-backend-production.up.railway.app`).

---

## 2. Frontend Deployment (SiteGround)

SiteGround is an excellent host for static files generated by Vite. For this project, the intended production domain is **csun.sose.dev**.

### Preparation Steps

1. **Environment Configuration:**
   Create a `.env.production` file in the `frontend/` directory to store the production API URL obtained from Railway.

   ```env
   VITE_API_BASE_URL=https://pdf2qti-backend-production.up.railway.app
   ```

   _Note: Ensure your `axios` configuration in the frontend code is utilizing this environment variable instead of a hardcoded `http://127.0.0.1:8000`._

2. **Build the Application:**
   Run the Vite build process locally (or via a CI/CD pipeline).
   ```bash
   cd frontend
   npm install
   npm run build
   ```
   This will bundle all assets and generate a `dist/` directory containing the optimized static files.

### Deployment Steps

1. Log in to your SiteGround account and access the Site Tools for the domain managing **sose.dev**.
2. Navigate to **Domain > Subdomains** and ensure `csun.sose.dev` is created and pointing to the correct directory.
3. Navigate to **Site > File Manager**.
4. Open the root directory for `csun.sose.dev`.
5. Upload all files and folders located _inside_ your local `frontend/dist/` directory directly into that folder.
6. **Handling SPA Routing (Optional):**
   If React Router is added in the future, create an `.htaccess` file in `public_html/` with the following fallback rules to ensure page refreshes route back to `index.html`:
   ```apache
   <IfModule mod_rewrite.c>
     RewriteEngine On
     RewriteBase /
     RewriteRule ^index\.html$ - [L]
     RewriteCond %{REQUEST_FILENAME} !-f
     RewriteCond %{REQUEST_FILENAME} !-d
     RewriteRule . /index.html [L]
   </IfModule>
   ```

## Final Verification

Once both services are deployed, navigate to your SiteGround domain. Upload a test PDF and verify that the backend successfully processes it and returns the downloadable QTI file. If errors occur, monitor the Railway logs for backend tracebacks and the browser's developer console for CORS or network issues.
