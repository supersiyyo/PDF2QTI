# Deployment & Migration Guide

This guide details the steps required to migrate the CSUN x SOSE Educational Tools Platform from a local development environment to production, utilizing **Railway** for the backend APIs and **SiteGround** for the React frontend.

## 1. Backend Deployment (Railway)

Railway provides a seamless deployment experience for Python applications by automatically detecting the environment via Nixpacks and installing dependencies from `requirements.txt`.

### Preparation Steps

1. **Configure CORS:**
   In `backend/main.py`, the `CORSMiddleware` is configured to allow requests from the production domain: **https://csun.sose.dev**.

   ```python
   allow_origins=["http://localhost:5173", "http://127.0.0.1:5173", "https://csun.sose.dev"]
   ```

2. **Define Start Command:**
   Create a file named `Procfile` in the `backend/` directory with the following content:
   ```text
   web: uvicorn main:app --host 0.0.0.0 --port $PORT
   ```

### Deployment Steps

1. Push your repository to GitHub.
2. Log in to Railway and create a new project.
3. Choose "Deploy from GitHub repo" and select your repository.
4. Go to the Service Settings and configure the **Root Directory** to `/backend`.
5. Navigate to the Variables tab and add your `GEMINI_API_KEY`.
6. Note the Public Networking URL generated for the backend (e.g., `https://pdf2qti-backend-production.up.railway.app`).

## 2. Frontend Deployment (SiteGround)

SiteGround hosts the static files generated by Vite. The production domain is **csun.sose.dev**.

### Preparation Steps

1. **Environment Configuration:**
   Create a `.env.production` file in the `frontend/` directory to store the production API URL.

   ```env
   VITE_API_BASE_URL=https://pdf2qti-backend-production.up.railway.app
   ```

2. **Build the Application:**
   Run the Vite build process locally:
   ```bash
   cd frontend
   npm install
   npm run build
   ```
   This generates a `dist/` directory containing the optimized static files.

### Deployment Steps (Automated via GitHub Actions)

We use a GitHub Actions workflow (`.github/workflows/deploy-frontend.yml`) to automatically build and deploy the frontend via FTP on every push to `main`.

**Required GitHub Secrets:**
1. `FTP_SERVER`: Your SiteGround FTP host (e.g., `ftp.csun.sose.dev`).
2. `FTP_USERNAME`: The FTP account name.
3. `FTP_PASSWORD`: The FTP account password.

### 3. Handling Large Binaries (Canvas Critter)

The repository `.gitignore` ignores large standalone executables like the 74.4MB `mostrecentCancanbuild90percentdone.zip` file for the Canvas Critter tool. This prevents repository bloat.
To deploy this binary:
1. Ensure the Windows build is compressed as `mostrecentCancanbuild90percentdone.zip`.
2. Access the SiteGround File Manager or connect via FTP.
3. Create a `downloads/` folder inside `public_html/`.
4. Upload the zip file into the `downloads/` directory. 
5. The React frontend will automatically resolve the link `https://csun.sose.dev/downloads/mostrecentCancanbuild90percentdone.zip`.

### 4. Handling React Router (SPA Routing)

Because the platform uses `react-router-dom` to manage multiple tool routes (e.g., `/instructor`, `/student`), an `.htaccess` file is critical. It must be placed in the `public_html/` directory on SiteGround to route all sub-paths back to `index.html`:

```apache
<IfModule mod_rewrite.c>
  RewriteEngine On
  RewriteBase /
  RewriteRule ^index\.html$ - [L]
  RewriteCond %{REQUEST_FILENAME} !-f
  RewriteCond %{REQUEST_FILENAME} !-d
  RewriteRule . /index.html [L]
</IfModule>
```

## Final Verification

Once deployed, navigate to **https://csun.sose.dev**, upload a test PDF, and verify that the backend processes it and returns a QTI file. Monitor Railway logs for any server-side issues.
